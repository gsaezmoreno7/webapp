Auditoría XSS

Demostración de Cross-Site Scripting y su mitigación

VULNERABLE

💀 Sistema de Comentarios — Versión VULNERABLE

Esta versión usa innerHTML para renderizar comentarios. Cualquier código HTML/JS inyectado será ejecutado por el navegador.

🎯 Payload de prueba XSS:

<img src=x onerror=alert(1)>

Comentarios (renderizados con innerHTML):

⚠️ Código VULNERABLE utilizado:

// ❌ PELIGROSO — Permite ejecución de código
contenedor.innerHTML += `<div>${comentario}</div>`;

// El navegador interpreta el HTML y ejecuta scripts
// como <img src=x onerror=alert(1)>
SEGURO

🛡️ Sistema de Comentarios — Versión SEGURA

Esta versión usa innerText / textContent para renderizar comentarios. Todo el contenido se trata como texto plano, neutralizando cualquier código inyectado.

Comentarios (renderizados con innerText):

✅ Código SEGURO utilizado:

// ✅ SEGURO — Escapa automáticamente el HTML
const div = document.createElement('div');
div.textContent = comentario; // o div.innerText
contenedor.appendChild(div);

// El navegador muestra el texto literal:
// "<img src=x onerror=alert(1)>" (como texto, no ejecuta)

📋 Mini-Informe: ¿Por qué NO confiar en el cliente?

1. El navegador ejecuta todo lo que recibe

Cuando se usa innerHTML, el navegador interpreta el string como código HTML real. Si un atacante inyecta <script>, <img onerror>, o <svg onload>, el navegador lo ejecutará sin cuestionarlo.

2. El input del usuario es una superficie de ataque

Cualquier campo de entrada (formularios, URLs, cookies, headers HTTP) puede ser manipulado por un atacante. Nunca se debe confiar en datos del cliente sin sanitización.

3. Consecuencias del XSS

  • 🔓 Robo de sesión: document.cookie puede ser enviado a un servidor externo
  • 👤 Suplantación de identidad: El atacante actúa como el usuario víctima
  • 🔄 Redirección maliciosa: window.location puede llevar a sitios de phishing
  • 📝 Modificación del DOM: El atacante puede alterar la interfaz visible
  • 🔑 Keylogging: Captura de teclas mediante listeners inyectados

4. Mitigaciones correctas

  • ✅ Usar textContent o innerText en vez de innerHTML
  • ✅ Sanitizar inputs en el servidor (nunca solo en el cliente)
  • ✅ Implementar Content Security Policy (CSP) headers
  • ✅ Usar funciones de escape como DOMPurify.sanitize()
  • ✅ Validar y escapar datos en ambos lados (cliente + servidor)
